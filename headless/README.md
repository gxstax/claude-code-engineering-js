# 思考题
1. 你的团队有哪些重复性的代码审查任务可以用 Headless 模式自动化？试着列出三个具体场景，并为每个场景设计  --allowedTools  和  --max-turns  的参数组合。
   > 目前只想到代码审核这一个
   > --allowedTools: Read, Glob, Grep
   > --max-turns: 3

2. 如果 Claude 在 CI 中自动修复了一个 Bug，但修复方式不是最优的，你会如何处理？
   > 重新拉一个pr，让人工介入处理
   
3. 请从流程设计的角度思考，如何在“自动化效率”和“代码质量控制”之间找到平衡点。
   > 指定代码质量的最低容忍度规则，规则检测通过则自动提交，检测不通过则拒绝
   
4. Headless 模式不保存会话状态。如果你需要在多个 CI 步骤之间保持上下文（例如第一步分析代码，第二步根据分析结果生成测试），你会怎么设计？提示：考虑  --session-id  参数和文件系统中间状态的组合使用。
   
   
5. 对比 Pre-commit Hook 和 GitHub Actions 两种集成方式的优缺点。在什么场景下你会选择只用其中一种？在什么场景下你会同时使用两种？
   > Pre-commit Hook 更侧重于提交前的代码质量检测，它是一个前置性的代码安全检查；
   > GitHub Actions更侧重于自动化CI/CD，到了这一步说明代码质量基本没有太大的问题，这里即使有代码检查能力，也只是简单的语法扫描。
